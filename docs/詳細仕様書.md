# 請求書作成支援システム 詳細仕様書

**文書バージョン**: 1.0  
**作成日**: 2026年2月26日  
**対象システム**: 理容・美容施設の施術記録表を画像から読み取り、Excel請求書として出力するWebアプリケーション

---

## 1. 概要

### 1.1 目的

理容・美容施設で使用される**施術記録表**（紙またはPDF・画像）をアップロードし、OCR・記号識別・AIによる情報抽出を行い、編集・集計のうえ**Excel形式の請求書**として出力することを目的とする。

### 1.2 対象ユーザー

- 理容・美容施設の事務担当者
- 施術記録に基づいて請求書を作成する担当者

### 1.3 主な価値

- 手入力の削減と転記ミスの低減
- 記号（〇・×・チェック・斜線など）の自動判定
- 施設名・施術日・曜日などの自動抽出
- メニュー別人数集計と単価×人数による金額計算の自動化

## 2. システム構成

### 2.1 アーキテクチャ概要

```
[クライアント]  Next.js (React 19) SPA
       │
       ├── /api/analyze     → Azure Document Intelligence (OCR)
       ├── /api/gemini      → Google Gemini API (文書情報抽出)
       ├── /api/validate-menu → Gemini API (メニュー検証)
       ├── /api/export      → ExcelJS (請求書生成)
       └── /api/auth/*      → 認証（Cookie）
```

- **フロントエンド**: Next.js 16 App Router、React 19、TypeScript、Ant Design、Tailwind CSS
- **バックエンド**: Next.js API Routes（サーバーサイド）
- **外部API**: Azure Document Intelligence、Azure Custom Vision、Google Gemini API
- **認証**: Cookie ベース（`ocr_auth`）、proxy（旧 middleware）で保護

| 分類 | 技術 | 用途 |

## 3. 機能仕様

### 3.1 入力

| 項目 | 仕様 |
|------|------|
| 対応形式 | PDF、画像（複数ファイル可） |
| 複数枚 | 複数ページ・複数ファイルを一括アップロード可能 |
| PDF | クライアントで pdf.js によりページごとに画像化して処理 |
| 画像回転 | 各画像を90°単位で回転可能（解析前に反映） |

### 3.2 処理フロー（解析パイプライン）

1. **画像アップロード・プレビュー**
   - ファイル選択 → PDFの場合はページごとに画像化 → `PageImage[]` として保持
2. **「アップロード & 解析」実行**
   - 各画像に対して順次:
     - 回転を反映した Blob を生成
     - **Azure Document Intelligence** に送信（prebuilt-layout、api-version=2023-07-31）
     - ポーリングで結果取得（最大10回、1秒間隔）
     - 403/429 時はリトライ（最大5回、待機時間 5+2n 秒）
   - 各ページの `analyzeResult.content`（Markdown）を **Gemini API** に送信し、施設名・年・月・日・曜日を抽出
   - 検出されたテーブル（`rowCount + columnCount > 10`）に対して:
     - セルをY→X順にソート
     - **buildDisplayRows**: 氏名列・メニュー列・施術実施列を検出し、データ行を構築
     - 各データセルを画像でクロップし **Custom Vision** で分類（Circle/Check → 〇、Cross/Slash → × 等）
     - OCRテキスト（〇○✓✔︎レ、チェック、済、×✗等）も有無判定に利用
   - 2枚目以降で施設名・日付が無い場合は「同じ書類の続き」として1枚目の情報を適用
   - 複数テーブル結果を **mergeDisplayResults** でマージ（ヘッダーはX座標でソート、カットを先頭に固定）
   - マージ後に **validate-menu API** でメニュー候補を検証し、無効メニューを除外（顔そり・顔剃りは常に有効）
3. **表示**
   - 行: 氏名・メニュー列・施術実施
   - セル: 〇/×/空の表示とクリックでトグル編集

### 3.3 記号の自動識別

| 分類 | Custom Vision タグ | 扱い |
|------|---------------------|------|
| 有 | Circle, Check | 〇（施術あり） |
| 無 | Cross, Slash | ×（施術なし） |
| OCR補足 | 〇○✓✔︎レ、チェック、済 | 有とみなす |
| OCR補足 | ×✗、空 | 無とみなす |

- セル画像はテーブルの bounding polygon に基づき canvas でクロップし、Custom Vision の classify API に送信
- 同時実行数は 1（キューで順次実行）
- デバッグモード時はクロップ画像の Object URL をコンソールに出力可能

### 3.4 文書情報の自動抽出（Gemini）

- **入力**: Document Intelligence の `analyzeResult.content`（Markdown テキスト）
- **出力 JSON**: `facilityName`, `year`, `month`, `day`, `dayOfWeek`
- **ルール**:
  - 年の省略時: 現在が N 年 M 月なら、テキストの月 > M のとき N-1 年、それ以外は N 年
  - 曜日未記載時: 年・月・日から計算し「月」「火」「水」「木」「金」「土」「日」のいずれか1文字で返す
- **施設名**: フロア表記（2F, 3F, 1階 等）は除去して正規化

### 3.5 データの編集機能

| 機能 | 説明 |
|------|------|
| セルトグル | セルクリックで 〇 → × → 空 の順で切り替え |
| 列一括トグル | ヘッダークリックでその列のデータ行を一括で 〇/× 切り替え（施術実施列は除く） |
| 施術実施列一括編集 | 施術実施列ヘッダークリックでモーダルを表示。範囲: 全体 / 画像単位 / ページ単位。操作: 〇 / × / 反転 |
| メニュー列追加 | 「メニュー追加」で列を挿入（氏名・施術実施以外）。デフォルトは × |
| メニュー列削除 | 列ヘッダーの削除でメニュー列を削除（氏名・施術実施は不可） |
| 人の追加 | グループ（画像）単位で行を追加。氏名入力で新規行を追加 |
| 人の削除 | 行削除で該当行を削除（確認あり） |
| 値引き入力 | 各行に数値入力フィールドを追加。合計金額から差し引かれる |
| 備考入力 | 各行にテキスト入力フィールドを追加。請求書の備考欄に集約して表示 |

### 3.6 集計とExcel出力

- **集計確定**: 「施術実施」が 〇 の行のみを対象に、日付別・メニュー別に人数を集計
- **日付キー**: `docInfoByImage` から施設名・年・月・日・曜日を取得。同一画像グループは同じ日付として扱う
- **Excel出力**:
  - テンプレート: `public/templates/請求書テンプレート.xlsx`
  - タイトル: `御請求書(令和X年X月度)`（B6）
  - 施設名: B9（正規化済み施設名）
  - 日付: K2 に請求書作成日（令和X年X月X日）
  - メニュー: E18〜J18 にメニュー名（最大6列）、E20〜J20 に単価
  - 抽出価格 + `DEFAULT_MENU_PRICES` のフォールバック（カット 1800、カラー 3800 等）
  - データ行: 21, 23, 25, 27, 29 行に月・日・曜日・メニュー別金額・合計・備考（施術サマリ + 行備考の集約）
  - 合計金額: D14（各メニュー合計 - 日別合計値引き額）、総合計は K31:L32 の数式（SUM）
  - メニュー数は最大6個（Excelテンプレートの列制約）。`allowedMenus` でフィルタ可能
- **単価**: テンプレート・OCRから抽出した価格 + `DEFAULT_MENU_PRICES` のフォールバック（カット 1800、カラー 3800 等）

### 3.7 UI・表示

| 項目 | 仕様 |
|------|------|
| 画像プレビュー | 複数枚を横スクロール表示。ズーム 30〜300%。各画像で削除・90°回転・クリックで拡大モーダル |
| OCR結果テーブル | ズーム 50〜200%。スティッキーヘッダー、横スクロール同期。「OCRのみ表示」で画像非表示・テーブル拡大 |
| デバッグモード | 切り取りセル画像の Object URL をコンソール出力、テーブル構造ログ |
| 処理進捗 | 「処理中: N / M ページ」を表示 |
| 施設・日付表示 | 抽出済みの施設名・施術日を緑背景で表示 |
| 集計結果表示 | 日付別・メニュー別人数を青背景で表示（集計確定後） |

### 3.8 認証

- **方式**: Cookie `ocr_auth`（値 `"1"`）、httpOnly, sameSite: lax, path: /
- **ログイン**: POST `/api/auth/login` に `username`, `password`。固定値 `admin` / `admin` で成功時 Cookie 設定し `/` へリダイレクト
- **ログアウト**: POST `/api/auth/logout` で Cookie を削除
- **保護**: `proxy.ts`（Next.js 16 の proxy）で `/login` および `/api/auth/*` 以外をチェック。未認証時は API は 401、ページは `/login` へリダイレクト

## 4. API仕様

### 4.1 POST /api/analyze

- **役割**: Azure Document Intelligence（prebuilt-layout）でドキュメント解析
- **入力**: `multipart/form-data`、`file`: 1ファイル（PDFまたは画像）
- **処理**: 分析開始 → Operation-Location でポーリング（1秒間隔、最大10回）
- **レスポンス**: Azure の analyze 結果 JSON（`analyzeResult.tables`, `analyzeResult.content` 等）
- **環境変数**: `AZURE_DI_ENDPOINT`, `AZURE_DI_KEY`

### 4.2 POST /api/gemini

- **役割**: テキストから施設名・年・月・日・曜日を抽出
- **入力**: `{ "text": string }`
- **出力**: `{ facilityName, year, month, day, dayOfWeek }`（JSON）
- **環境変数**: `GEMINI_API_KEY`, `GEMINI_MODEL_NAME`（省略時 gemini-2.5-flash）, `GEMINI_ENDPOINT`（任意）

### 4.3 POST /api/validate-menu

- **役割**: メニュー候補が実際の施術メニューかどうか判定（Gemini）
- **入力**: `{ "menuCandidates": string[] }`
- **出力**: `{ validMenus: string[], invalidMenus: string[] }`
- **環境変数**: `GEMINI_API_KEY`, `GEMINI_MODEL_NAME`

### 4.4 POST /api/export

- **役割**: 集計データをExcel請求書として出力
- **入力**: `{ dateDataList: DateData[], allowedMenus?: string[] }`
  - `DateData`: `{ counts: Record<string, number>, unitPrices?: Record<string, number>, totalDiscount?: number, remarks?: string, facility?, reiwaYear?, month?, day?, weekday? }`
- **出力**: Excel バイナリ（`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`）、ファイル名 `order.xlsx`
- **依存**: `public/templates/請求書テンプレート.xlsx` が存在すること

### 4.5 POST /api/auth/login

- **入力**: `{ username: string, password: string }`
- **成功**: 200, Cookie `ocr_auth=1` 設定
- **失敗**: 401

### 4.6 POST /api/auth/logout

- **処理**: Cookie `ocr_auth` を削除（maxAge: 0）
- **レスポンス**: 200 `{ ok: true }`

## 6. 外部サービス連携

### 6.1 Azure Document Intelligence

- **モデル**: prebuilt-layout
- **APIバージョン**: 2023-07-31
- **レート制限**: 画像間 1 秒待機。403/429 時はクライアントでリトライ（最大5回）
- **環境変数**: README では `DOCUMENT_INTELLIGENCE_*`、実装は `AZURE_DI_ENDPOINT`, `AZURE_DI_KEY`

### 6.2 Azure Custom Vision

- **用途**: テーブルセル画像の分類（Circle / Check / Cross / Slash 等）
- **呼び出し**: クライアントから直接 `customvision/v3.0/Prediction/{projectId}/classify/iterations/{iterationId}/image`
- **環境変数**: `NEXT_PUBLIC_CUSTOM_VISION_KEY`, `NEXT_PUBLIC_CUSTOM_VISION_ENDPOINT`, `NEXT_PUBLIC_CUSTOM_VISION_PROJECT_ID`, `NEXT_PUBLIC_CUSTOM_VISION_ITERATION_ID`

### 6.3 Google Gemini API

- **用途**: 文書情報抽出（/api/gemini）、メニュー検証（/api/validate-menu）
- **モデル**: gemini-2.5-flash（既定）。v1beta で JSON スキーマ指定可能
- **環境変数**: `GEMINI_API_KEY`, `GEMINI_MODEL_NAME`, `GEMINI_ENDPOINT`（任意）

## 7. 制限事項・既知の制約

- **メニュー列数**: Excel 出力時は最大 6 メニュー（テンプレートの列数による）
- **日付データ行**: Excel テンプレートのデータ行は 5 件分（21, 23, 25, 27, 29 行）
- **Azure DI**: Free Tier (F0) 使用時はクォータ制限に注意。ポーリングは最大 10 回
- **認証**: ユーザー名・パスワードはコード内固定（admin/admin）。本番では環境変数化やID連携を推奨
- **テンプレート**: `public/templates/請求書テンプレート.xlsx` が無いと Excel 出力は 500 エラー
- **氏名列**: テーブルに「氏名」を含むセルが無いとそのテーブルはスキップ（フォールバックで前回のヘッダーを使う場合は継続可能）
- **山田太郎行**: サンプル行として検出された場合はデータ行から除外

## 10. 用語集

| 用語 | 説明 |
|------|------|
| 施術記録表 | 氏名・メニュー（カット/カラー等）・施術有無（〇×）を記録した表 |
| 施術実施 | その日その人が施術を受けたかどうかを示す列（集計対象の判定に使用） |
| 日付キー | 集計・Excelの日付単位のキー（例: 2026-2-15、不明-0） |
| グループID | 同一画像（同一テーブル）に属する行の groupId。人の追加・一括編集の範囲に使用 |
| フォールバックモード | 氏名列が検出できなかった場合、直前のテーブルのヘッダー構成を流用するモード |
